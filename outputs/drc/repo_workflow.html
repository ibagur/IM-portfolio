Summary
  • You do not need to run any scripts. All processed outputs are already committed.
  • Use build/drc_health_zones.geojson for mapping / GIS — geometry + all 25 vector layers in one file.
  • Use build/long/<dataset>__<metric>.csv for flat tabular analysis (one row per zone per date).
  • OD matrices live in data/<dataset>/processed/ — find them via qa/matrix_log.csv.
  • Never use raw/ for analysis — those are untouched upstream sources.

For data consumers

You do not need to run any scripts. Pull the repo and use the committed outputs directly.

Recommended outputs

OutputFormatBest for
build/drc_health_zones.geojson GeoJSON Mapping & GIS — all 25 vector layers embedded per zone, ready for QGIS, geopandas, sf, or web maps
build/long/<dataset>__<metric>.csv Long-format CSV Tabular analysis — one row per zone (per date for time-series), canonical zone names resolved
data/<dataset>/processed/*.csv CSV Same as build/long/ but per-dataset; useful before a build has been run locally
qa/matrix_log.csv + data/<dataset>/processed/*.matrix.csv OD matrix CSV Travel time, displacement flows, mobility — not embedded in GeoJSON
Never use raw/ for analysis. Raw files are upstream sources in their original format (PDFs, Excel workbooks, NetCDF rasters, zipped shapefiles).

Update frequency guide

🔴 Outbreak-linked (frequent) epi/, insp_sitrep/ — updated frequently during the active outbreak.
🟡 Static socioeconomic (annual) worldpop/, gdp_pc/, ccvi/, fao_lccs/ — only when a new upstream version is released.
🟢 Infrastructure (infrequent) grid3_healthsites/, healthsites_io/, refugee_sites/, osrm/ — on new GRID3 version or significant OSM update in DRC.
🟠 Mobility matrices (ad hoc) IDP/, flowminder/ — per IOM DTM round or Flowminder report; no fixed schedule.

ACLED conflict: placeholder at province grain. Zone-level event data requires a separate event-level pull — tracked in GitHub Issues.

For contributors — running scripts when raw data is updated

Scripts only need to be re-run when raw source files change. All Python commands use pipenv with Python 3.11. R scripts use your local R installation.

Pipeline overview

The dashed green line shows that build/long/ is a flat copy of processed/ — same data, different path. Matrices (dashed amber) bypass the GeoJSON merge; they stay in processed/ and are catalogued in qa/matrix_log.csv.

⚑ Standard contributor flow
1Update files in data/<dataset>/raw/
2Update metadata.yaml (at minimum: retrieved_on)
3Run the dataset's process script (see tables below)
4pipenv run python -m tools.qa <dataset> — fix any failures
5pipenv run python -m tools.build_geojson — only if vector outputs changed
6pipenv run python -m pytest tests/
7Open a PR
